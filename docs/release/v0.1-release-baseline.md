# AION Brain v0.1 Release Baseline

## Baseline Summary

AION Brain v0.1.0 is the local-first Brain baseline for AION OS, the Adaptive
Intelligence Orchestration Nexus Operating System.
AION means Adaptive Intelligence Orchestration Nexus.

## Included Subsystems

The baseline includes generic Brain contracts, event intake, policy, memory,
capability registry, module metadata, conformance, planning, runtime adapter
boundary, execution records, goals, tasks, schedules, reasoning, model gateway
boundary, prompt governance, response composition, grounding, beliefs,
entities, situations, decisions, outcomes, learning, reflection, skills,
audit, visual telemetry, operator overview, release candidate gate, freeze
gate, release package metadata, bootstrap, and golden path verification.

## Explicit Exclusions

- No production auth.
- No full autonomy.
- No external model calls by default.
- No external notification delivery by default.
- No extension code loading.
- No capability activation.
- No dynamic route registration.
- No hard delete.
- No model-generated tool execution.
- No domain modules in Brain core.

## Runtime Defaults

Runtime behavior is deterministic and local by default. Optional adapters remain
behind AION-owned boundaries and are disabled unless explicitly configured.

## Security Defaults

Secrets, private request metadata, private provider inputs, private reasoning
traces, provider payloads, and production credentials must not appear in
contracts, telemetry, logs, tests, examples, docs, or release evidence.

## Operator Defaults

Operator surfaces are read-mostly. Recommendations, alerts, queues, findings,
and handoff records do not remediate source issues automatically.

## Local Stack Defaults

The default stack is Brain API, Postgres, Redis, NATS, and OPA through Docker
Compose. Optional profiles remain outside the v0.1 release baseline.

## SDK/CLI State

The Python SDK and `aionctl` remain developer tooling for public Brain APIs.
They do not import server internals or bypass policy.

## Extension/Module State

Extensions and module bindings are metadata-only. They do not install packages,
load code, register routes dynamically, activate capabilities, or execute
module behavior in v0.1.

## Conformance/Readiness State

Conformance and readiness are schema and metadata gates. They may block future
activation, but they do not activate anything in v0.1.

## Post-v0.1 Unlocks

Post-v0.1 work may design governed activation, domain module integration, UI
operator console work, external connectors, deployment profiles, and provider
hardening only after the Brain baseline is tagged.

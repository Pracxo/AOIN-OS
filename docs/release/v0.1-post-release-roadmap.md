# AION v0.1 Post-Release Roadmap

1. AION-080 v0.1 local freeze and tag preparation
2. AION-081 post-tag integrity audit and CI hotfix stabilization
3. AION-082 module ecosystem strategy
4. AION-083 metadata-only activation request gate, still disabled
5. AION-084 generic knowledge module metadata package
6. AION-085 non-executing deterministic module mock runtime
7. AION-086 model provider adapter hardening
8. AION-087 operator UI decision
9. production auth design
10. deployment profile design

Module strategy starts after v0.1 Brain freeze.

Future vertical modules must plug into Brain through extension, binding,
conformance, action proposal, policy, audit, provenance, operator review,
sandbox, release, and freeze gates.

No domain module should modify Brain core.

Activation remains disabled until future design gates pass. Code loading
remains disabled. Package installation, dynamic route registration, full
autonomy, external calls, controlled handoff, and production deployment logic
remain disabled.

The first governed module class is Generic Knowledge Intelligence. AION-084
turns that selection into a metadata-only module pack under
`examples/modules/generic-knowledge-intelligence/`. Validate it with:

```bash
./scripts/module-pack-check.sh
./scripts/generic-knowledge-demo.sh --offline-ok --skip-api
```

The pack demonstrates retrieval, grounding, citation, summary, explanation,
and answer contracts without runtime activation risk.

## AION-086 Provider Readiness

AION-086 adds model provider adapter hardening, prompt egress guard, provider
readiness gate, and dry-run provider simulation. This remains readiness-only:
no provider SDKs, credentials, prompt transmission, external calls, model
invocation, or tool execution.

## AION-087 Operator Console Strategy

AION-087 stays CLI/API-first and creates the Operator Console strategy,
workflow map, future view spec, data safety rules, action boundaries, demo map,
no-go conditions, and future UI milestone plan.

It adds no runtime UI, no frontend dependencies, no API routes, no SDK
resources, no CLI commands, no migrations, no module activation, no code
loading, no package installation, no runtime registration, no external model
calls, and no external services.
